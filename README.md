# SkippyGames

Магазин игр (Steam / PlayStation / Xbox / Nintendo Switch) и внутриигровой
валюты на GitHub Pages, с автоматическим сбором данных из Steam.

## Структура

```
index.html                         — весь сайт (каталог + карточка игры + чат + лайтбокс)
data/index.json                    — лёгкий индекс каталога (для списка/поиска/фильтров)
data/games/{id}.json               — полные данные одной игры (описание, скриншоты, трейлер, DLC)
backend/parser.py                  — сборщик данных Steam -> data/index.json + data/games/*.json
backend/custom_ids.txt             — appid, которые нужно добавить вручную
backend/skipped_ids.json           — appid, которые парсер проверил и пропустил (не игры) — создаётся автоматически
backend/requirements.txt           — зависимости Python
.github/workflows/update_games.yml — автозапуск парсера каждые 6 часов
```

Сайт — одна страница (`index.html`), переключающая каталог и карточку игры
на чистом JavaScript. Ссылка на конкретную игру: `index.html?id=1245620`.

## Архитектура данных (важно понимать)

GitHub Pages — статический хостинг, там нет ни сервера, ни настоящей базы
данных. Поэтому "масштабируемая база данных" здесь реализована как
аккуратно спроектированная файловая структура:

- **`data/index.json`** — один файл со сводными данными ВСЕХ игр
  (id, название, обложка, цена, жанры, платформы). Именно он грузится
  целиком при заходе на сайт — при 10 000 игр это ~150 байт × 10 000 ≈
  1.5 МБ (а с учётом сжатия, которое GitHub Pages включает автоматически,
  на практике в разы меньше), что вполне приемлемо для одного запроса.
- **`data/games/{id}.json`** — по одному файлу на игру с "тяжёлыми" данными
  (полное описание, скриншоты в высоком разрешении, трейлер, DLC). Такой
  файл подгружается лениво (`fetch`) только в момент, когда пользователь
  открывает карточку конкретной игры — а не все 10 000 сразу.

Это и есть реализация пунктов "пагинация, ленивая загрузка, эффективная
индексация" в условиях статического хостинга без бэкенда.

## Как это работает

1. `backend/parser.py` запускается по расписанию через GitHub Actions —
   каждые 6 часов (можно и вручную: Actions → "Update game catalog" → Run
   workflow).
2. Каждый прогон работает инкрементально:
   - обновляет цены/данные у самых "старых" карточек в каталоге (не
     обновлялись дольше 3 дней, до 1200 штук за прогон);
   - добавляет новые игры — сначала из `backend/custom_ids.txt` и топ
     продаж Steam, затем из полного списка приложений Steam, пока не
     будет достигнута цель в 10 000 игр или не закончится бюджет времени
     (55 минут за прогон, до 600 новых игр);
   - appid, которые оказались не играми, запоминаются в
     `backend/skipped_ids.json`, чтобы не тратить на них запросы повторно.
3. Экшен коммитит изменения в `data/` и `backend/skipped_ids.json` обратно
   в репозиторий.
4. `index.html` сначала загружает `data/index.json`, а полные данные
   конкретной игры — лениво, из `data/games/{id}.json`, только при её
   открытии.

### Почему каталог растёт постепенно, а не мгновенно

У Steam Store API нет официального ключа для больших объёмов запросов, и
он ограничивает частоту обращений с одного IP. Чтобы не получить временную
блокировку и уложиться в лимит GitHub Actions (6 часов на job), парсер
добавляет до 600 новых игр за прогон. При запуске каждые 6 часов каталог
реалистично дорастает до 10 000+ игр за 1–2 недели, дальше поддерживается
в актуальном состоянии.

## Реализованные исправления и новые возможности

### 1. Чат ВКонтакте

Виджет — официальный `VK.Widgets.CommunityMessages`. Если раньше он
открывался пустым окном, то теперь код:

- показывает индикатор загрузки, пока виджет монтируется;
- ставит таймаут (6 секунд) и проверяет, появился ли внутри контейнера
  `<iframe>`, который создаёт сам VK;
- если iframe не появился — показывает понятную заглушку с прямыми
  ссылками на VK и Telegram и кнопкой "Попробовать снова", а не оставляет
  пустое окно.

**Важно:** пустое окно виджета почти всегда вызвано не кодом, а
настройками самого сообщества ВКонтакте. Проверьте:

1. Управление сообществом → Сообщения → включена галочка "Разрешить
   использование виджета".
2. В том же разделе, в поле **"Разрешённые домены"**, добавлен точный
   домен вашего сайта на GitHub Pages (например `username.github.io`, без
   `https://` и без завершающего слэша).
3. Виджет один раз создан через генератор https://vk.com/dev/CommunityMessages
   (это привязывает числовой ID сообщества к виджету на стороне VK).

Без этих трёх шагов НИКАКОЙ код не заставит виджет VK отобразиться —
проверка домена происходит на стороне серверов VK, а не в браузере
пользователя.

### 2. Трейлеры: фолбэк на YouTube

Раньше при отсутствии трейлера в Steam показывалась заглушка. Теперь
`backend/parser.py`, если у игры нет трейлера в Steam, один раз ищет
подходящее видео через официальный **YouTube Data API v3** и сохраняет
`trailer_youtube_id` в `data/games/{id}.json`. На сайте это видео
показывается в отзывчивом (responsive) `<iframe>`.

**Настройка (обязательно для этой функции):**

1. Получите бесплатный API-ключ: Google Cloud Console → создать проект →
   включить "YouTube Data API v3" → Credentials → Create API key.
   Ограничьте ключ по HTTP-referrer или API, чтобы его нельзя было
   использовать с других сайтов.
2. В репозитории: Settings → Secrets and variables → Actions → New
   repository secret → имя `YOUTUBE_API_KEY`, значение — ваш ключ.
3. Готово — workflow уже передаёт этот секрет парсеру автоматически.

**Важно про квоту:** YouTube Data API даёт 10 000 "units"/день бесплатно,
а один поисковый запрос стоит 100 units — то есть с одного ключа доступно
максимум ~100 новых поисков трейлеров в сутки. Именно поэтому поиск
трейлера сделан на бэкенде (один раз при сборе данных), а не в браузере
каждого посетителя сайта — иначе квота исчерпывалась бы за несколько
первых посетителей. Если ключ не задан или лимит исчерпан — сайт честно
покажет сообщение "Трейлер не найден" со ссылкой "Искать на YouTube"
вместо того, чтобы показывать случайное видео.

### 3. Полный каталог Steam

См. раздел "Архитектура данных" выше. Ключевые технические решения:

- Инкрементальный сбор данных (не пытаемся забрать все игры за один
  прогон — это физически невозможно из-за rate-limit Steam).
- Разделение "лёгких" данных индекса и "тяжёлых" данных карточки игры.
- На фронтенде: пагинация (24 игры на страницу), debounce поиска (200 мс),
  `loading="lazy"` для всех изображений, кэш уже загруженных карточек игр
  в памяти (`gameDetailCache`), чтобы повторное открытие не делало
  повторный запрос.

## Настройка перед запуском

### 1. Числовой ID группы ВКонтакте

В `index.html`, в начале блока `<script>`:

```js
var VK_GROUP_NUMERIC_ID = 195484236;
```

### 2. YouTube Data API (для фолбэка трейлеров)

См. раздел "Трейлеры: фолбэк на YouTube" выше.

### 3. Кастомный список игр и платформ

Добавляйте appid построчно в `backend/custom_ids.txt`. Если игра есть на
PlayStation/Xbox/Switch — отредактируйте `"platforms"` в соответствующем
`data/games/{id}.json` вручную (Steam не сообщает о консольных релизах).

### 4. Аватарки дежурного менеджера

Массив `MANAGER_AVATARS` в `index.html` — 7 рабочих ссылок-заглушек
(placehold.co), замените на реальные изображения, когда будут готовы.

## Деплой на GitHub Pages

1. Загрузите все файлы проекта в репозиторий (ветка `main`).
2. Settings → Pages → Source → ветка `main`, папка `/ (root)`.
3. Settings → Actions → General → Workflow permissions → "Read and write
   permissions".
4. (Опционально, для трейлеров) Settings → Secrets and variables → Actions
   → добавить `YOUTUBE_API_KEY`.
5. Запустите workflow вручную первый раз: Actions → "Update game catalog"
   → Run workflow.
6. Откройте `https://<ваш-логин>.github.io/<репозиторий>/` и добавьте этот
   домен в "Разрешённые домены" виджета VK (см. раздел про чат выше).

## Локальный запуск парсера

```bash
cd backend
pip install -r requirements.txt
export YOUTUBE_API_KEY=ваш_ключ   # необязательно, но нужно для трейлеров
python parser.py
```
